# 后端架构审查

这是对当前后端的一次“是否足够精简、还有哪些地方可优化”的审查。

## 总体结论

**整体方向是对的，而且已经足够支撑你当前这版前端。**

优点：
1. 控制核心仍然留在 C++ daemon
2. Python 层只做前端适配，没有把电机逻辑搬上来
3. 单电机调试、回放控制、状态推流三条线已经分清
4. 命令协议保持简单，前端容易接
5. 你要求保留的安全策略仍在，没有乱加新保护

---

## 我认为目前做得好的地方

### 1. `TwinAgent` 是清晰的系统边界
它把：
- 网络
- motion
- replay
- 状态快照
集中到一个文件里，符合当前项目规模。

### 2. `MotorIO` 的抽象是对的
上层看的是 relative joint target，而不是直接碰电机原始位置。

### 3. Python 层分层合理
- `RobotClient`
- `RobotBackend`
- `ReplayController`
- `ExperimentManager`

这正适合给 Qt 前端复用。

---

## 我认为还可以优化的点

## A. 版本包里不该带 `__pycache__`
这是发布产物洁净度问题。  
下次打包建议排除：
- `__pycache__/`
- `*.pyc`

### 建议优先级
低，但应该修。

---

## B. 状态推流目前只支持单客户端串行连接
当前 `StateLoop()` 是：
- `accept` 一个客户端
- 持续推给它
- 直到断开后才接下一个

这意味着：
- Qt 前端连上后，CLI 或别的监控器不能再同时订阅

### 是否现在必须改
**不是必须。**  
因为你这版前端大概率会独占状态流。

### 后续改法
把状态流改成：
- 一个 accept 线程
- 多个 client fd 列表
- 广播发送
- 断开时移除

---

## C. 命令口更适合单客户端长期持有
现在命令通道本身没错，但更适合：
- 前端一个长期连接
- 不要 GUI、CLI、脚本多方同时抢控制权

### 是否现在必须改
不是必须。  
靠使用约束就能工作。

---

## D. `StartReplayMotion()` 会复制整份 replay 样本
`replay_samples_` 被复制到 lambda capture。  
小文件无所谓，大文件会多一次内存复制。

### 是否现在必须改
不是必须。  
如果你的 CSV 量级只是几千到几万帧，这仍可接受。

### 更优做法
改成：
- `shared_ptr<const std::vector<ReplaySample>>`
- motion 线程只持有共享引用

---

## E. Python import 还是脚本风格
现在有很多：
- `from robot_backend import RobotBackend`
- `from recorder import ReplayCsvStore`

这适合“平铺脚本目录”，但不利于以后做 pip package。

### 是否现在必须改
不是必须。  
你当前直接跑 PySide6 项目完全够用。

### 后续建议
如果以后要做成真正包，改为：
- `from .robot_backend import RobotBackend`

---

## F. `joint_torques` 字段命名可以更明确
当前状态里叫 `joint_torques`，但它来自底层 `motor_tau`。  
这在调试里没问题，但在严格语义上容易让前端误解成“已完成关节侧换算”。

### 是否现在必须改
不建议立刻改字段名，因为前端已经要对这个字段对接了。  
更稳妥的是：
- 先在文档里明确语义
- 以后真要换算，再新增 `motor_torques` / `joint_torques_est`

---

## G. `ExperimentManager` 还很薄
这不算坏事。  
但以后前端一复杂，可能会需要它承担：
- 自动重连
- 错误码到 UI 文案映射
- 上传任务状态
- 心跳与连接健康

### 是否现在必须改
不是。  
当前保持薄层是合理的。

---

## 当前是否“足够精简”

我的判断是：

**是，已经足够精简，可以先做前端。**

原因：
- 真机链路没有被 UI 污染
- 单电机和 replay 已经是最核心的最小闭环
- Python 只是胶水层，没有重复实现控制逻辑
- 没有额外引入复杂框架

所以当前不建议大动后端架构，只建议做**小范围清理**。

---

## 最推荐的下一步

1. 用这版协议先把 Qt 前端做出来
2. 真机联调跑通
3. 再决定是否要做以下两项升级：
   - 多客户端状态广播
   - Python 包化与更正式的 service 层
